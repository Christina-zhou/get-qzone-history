# QQ空间说说导出工具：AI 项目说明手册

这份文档是写给 AI 助手、代理式编码工具和后续维护者的。  
目标是让 AI 在不反复踩坑的前提下，正确使用和维护这个项目。

## 1. 项目目标

本项目用于导出 QQ 空间历史说说，并生成：

- `json`
- `xlsx`
- `html`
- `checkpoint`
- 本地图片目录

当前稳定主路径是：

- 使用真实浏览器登录
- 抓取桌面版 Qzone 说说 iframe
- 在列表页内展开长文本
- 逐页点击“下一页”
- 下载图片并生成本地浏览页

## 2. 当前应坚持的技术路线

### 正确路线

- 桌面版 DOM 抓取
- 列表页展开全文后抓正文
- 通过“下一页”继续翻页
- 下载图片时直接访问图片 URL
- 长任务使用 checkpoint

### 不应再回退的路线

- 不要再把“长文本补全”做成访问详情页
- 不要再把分页主逻辑改成页码跳转推进
- 不要把图片提取范围扩到整个卡片的所有 `href/background-image`
- 不要把代码里的中文文案改坏成乱码

## 3. 当前关键文件

### 核心脚本

`get_qzone_history_browser.js`

职责：

- 打开浏览器
- 登录
- 进入桌面版或移动版页面
- 分页抓取
- 图片下载
- 写出 `json/xlsx/html/checkpoint`

### Python 包装入口

`get_qzone_history.py`

职责：

- 只是转调 Node 脚本
- 不承载真正业务逻辑

### 配置文件

`qzone_history.ini`

### 用户说明

`README.md`

## 4. 当前稳定行为

### 分页

当前分页正确策略：

1. 从桌面版说说 iframe 抓当前页
2. 点击中文“下一页”
3. 等待页面刷新
4. 再抓下一页

注意：

- “下一页”文案必须保持中文
- 之前出现过 bug，是因为相关中文被写成乱码，导致选择器匹配失败

### 长文本

当前正确策略：

- 只在列表页展开
- 展开按钮文案匹配：
  - `全文`
  - `展开`
  - `更多`

禁止：

- 重新访问详情页补正文

原因：

- 详情页结构不稳定
- 会显著增加请求数
- 容易触发风控
- 之前已验证不是稳定方案

### 图片

当前正确策略：

- 只抓正文图片区的 `img.src`
- 不要扩展到 `data-src`、`href`、`background-image`
- 下载优先使用真实图片 URL
- 下载失败再使用页面态或浏览器导航兜底

## 5. 当前已经踩过的坑

### 坑 1：中文被改成乱码

后果：

- “下一页”识别失效
- “全文/展开/更多”识别失效
- “请先登录”判断失效
- HTML 成品中的中文说明变脏

规避要求：

- 编辑文件时必须使用 UTF-8
- 不要用会重写编码的脚本式覆盖写法
- 修改中文字符串时，改动后要全文搜索检查

建议检查词：

- `下一页`
- `全文`
- `展开`
- `更多`
- `请先登录`
- `QQ空间说说浏览`

### 坑 2：分页误判结束

现象：

- 明明还有后续页
- 日志只到第 2 页
- 随后直接进入导出阶段

根因：

- “下一页”识别失败

当前要求：

- 分页推进必须以“下一页”识别为主

### 坑 3：详情页补全文字带来副作用

问题：

- 请求数上升
- 可能限流
- 详情页结构不稳定
- 详情页图片提取容易误抓头像等资源

结论：

- 已废弃
- 不要重新加回

### 坑 4：图片过抓

错误做法：

- 扫整张卡片所有图片
- 扫 `href`
- 扫 `background-image`

后果：

- 头像、装饰图、无关资源混入

正确做法：

- 只抓正文图片区的 `img.src`

## 6. 推荐的 AI 工作流程

### 当用户说“开始抓”

优先执行：

1. 检查脚本语法
2. 启动正式任务
3. 用日志文件持续观察
4. 定期向用户回报页数和累计条数

### 当用户说“查进度”

优先查看：

- 正在运行的 `node` 进程
- 最新日志
- 最新 `checkpoint.json`

应返回：

- 当前页数
- 当前累计条数
- 是否在暂停
- 是否仍在运行

### 当用户说“修 bug”

优先顺序：

1. 先看日志
2. 再看当前代码相关函数
3. 尽量做最小修改
4. 先小跑验证
5. 再正式长跑

### 当用户说“继续抓”

优先考虑：

- 是否应从最新 checkpoint 恢复
- 是否要继续上次的页码范围
- 是否需要降低速度防止限流

## 7. 推荐测试策略

### 小样验证

优先跑：

- `--start-page 1 --end-page 5`

验证点：

- 是否能翻过第 2 页
- 是否能正常展开长文本
- 图片是否下载
- HTML 是否可打开

### 正式抓取

推荐参数：

```bash
node get_qzone_history_browser.js ^
  --start-page 1 ^
  --end-page 200 ^
  --page-wait-ms 20000 ^
  --segment-pages 10 ^
  --segment-pause-ms 180000 ^
  --stagnant-page-limit 20
```

## 8. 输出结果约定

每次成功运行后，通常会生成：

- `<qq>_<timestamp>.json`
- `<qq>_<timestamp>.xlsx`
- `<qq>_<timestamp>.html`
- `<qq>_<timestamp>.checkpoint.json`
- `<qq>_<timestamp>_assets/images/`

AI 在汇报时，应该优先告诉用户：

- 一共多少条
- 到第几页
- 图片下载多少张
- 最新 `html` 路径

## 9. AI 修改代码时的硬性约束

- 不要重新引入详情页补全文字逻辑
- 不要把分页主逻辑切回页码跳转推进
- 不要修改中文文案为乱码
- 不要为了抓图片而放宽到全页面资源
- 不要在未验证前直接启动大规模抓取

## 10. AI 可直接复用的说明模板

当 AI 需要向用户说明当前抓取状态时，建议用这类格式：

```text
当前进度正常：
- 已抓到第 X 页
- 当前累计 X 条唯一说说
- 进程仍在运行
- 当前处于第 X 段暂停期 / 正在继续抓取

最新检查点：
resource/result/xxx.checkpoint.json
```

当 AI 需要说明异常时，建议用这类格式：

```text
这轮没有继续推进，原因不是登录失败，而是分页没有跨到下一页。
当前结果停在：
- 第 X 页
- 共 X 条

下一步需要直接检查桌面版 iframe 中的分页 DOM，再修“下一页”识别。
```
